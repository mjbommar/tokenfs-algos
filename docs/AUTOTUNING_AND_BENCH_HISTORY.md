# Autotuning And Bench History

Date: 2026-04-30.

The workload matrix gives us evidence. Autotuning turns that evidence into
dispatch decisions. The benchmark history keeps those decisions auditable across
commits, compiler versions, and processors.

## Logging Contract

Every workload benchmark run should record:

- git commit and dirty flag;
- rustc version;
- target triple and enabled feature family when available;
- CPU model;
- selected backend;
- cache hierarchy;
- logical CPU count;
- workload axes;
- measured entropy summaries;
- chosen kernel or planner recommendation;
- Criterion mean time and throughput.

The current history files live under `target/bench-history/`. For long-lived
tracking, set `TOKENFS_ALGOS_BENCH_HISTORY` to a persistent directory outside
`target`.

## Calibration Plan

The first calibration pass should stay short and deterministic:

| Probe | What it learns |
|---|---|
| 1 MiB high-entropy histogram | General counting throughput. |
| 1 MiB zeros histogram | Dependency-chain and run-detection behavior. |
| 4 KiB sequential loop | Small-read classifier overhead. |
| 64 KiB sequential loop | File-chunk behavior. |
| Macro-mixed file slice | Region planning value. |
| Random 1-byte calls | API fixed overhead. |
| Parallel 64 KiB scan | Thread-local/reduction cost. |

The result should be cached by a key roughly equivalent to:

```text
crate version + binary hash + rustc version + backend + cpu model + cache profile
```

The cache can start as JSON. It does not need to be part of the public API until
the planner has enough evidence to be stable.

## Dispatch Table Shape

A future generated dispatch table can look like:

```text
primitive=histogram
backend=avx2
api_context=sequential
content=binary
entropy=high
scale=flat
chunk_bucket=4k
kernel=adaptive-prefix-1k
confidence=observed
```

Initially this table should be hand-authored from benchmark history. Later it
can be generated by an xtask from accumulated JSONL rows.

## Interpreting Results

Do not compare all rows as if they mean the same thing:

- `block` rows measure whole-buffer algorithm throughput.
- `sequential` rows measure repeated-call behavior over the whole buffer.
- `random` 4 KiB rows approximate page-like random reads.
- `random` 1-byte rows are overhead stress tests, not bulk throughput.
- `parallel-sequential` rows measure scheduling, private state, and reduction.

The planner should use this context before considering ISA or cache details.

## Current Tooling Slice

The first processor-aware slice is implemented:

- benchmark JSONL rows include a processor profile with architecture, OS,
  detected CPU features, logical CPU count under the current process affinity,
  and Linux cache topology when available;
- `ProcessorProfile::detect()` also reports Linux cache line, L1D, L2, and L3
  sizes inside the library when `std` is enabled;
- workload manifests include planner recommendations and explanations;
- the latest Markdown summary shows the planner choice beside the measured best
  kernel for each workload row;
- `cargo xtask bench-compare <old.jsonl> <new.jsonl>` prints the largest
  matched throughput changes between two history runs;
- `cargo xtask bench-report [run.jsonl]` emits heatmap HTML, planner-vs-best
  and winner-count charts, thread-scaling charts, per-dimension cross-tab SVGs,
  a throughput histogram SVG, and timing CSV artifacts under
  `target/bench-history/reports/`;
- `cargo xtask bench-calibrate` runs a short adaptive workload matrix with
  topology thread rows enabled and logs it to benchmark history;
- named suites now cover smoke, synthetic-full, real ISO, F21/F22 paper data,
  size sweep, alignment sweep, thread topology, planner parity, cache hot/cold,
  and profile-oriented runs;
- `cargo xtask profile` stores `perf stat -d` output under `target/profiles/`
  when Linux `perf` is available;
- `cargo xtask profile-flamegraph` produces an SVG flamegraph when
  `cargo-flamegraph` is installed;
- `cargo run -p tokenfs-algos --example dispatch_explain` prints the detected
  processor profile, histogram catalog, and representative plans.

This is not an autotuner yet. It is the logging and planner substrate needed to
build one. For v0.1, the planner should remain static, transparent, and
feature-dispatch driven. A persistent calibration cache and generated dispatch
tables belong in v0.2, after the kernel set stabilizes.

## Near-Term Tooling

The next useful tooling increments are:

1. Write a compact calibration JSON summary from `bench-calibrate`, instead of
   only logging the underlying Criterion rows.
2. Teach `bench-compare` to group deltas by case/access/kernel and flag planner
   mismatches explicitly.
3. Feed calibration results back into the planner before falling through to
   static heuristics.
4. Add stable named Criterion baselines once scalar adaptive promotion starts.
