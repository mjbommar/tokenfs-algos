# USearch Deep Dive — Wire-Format & Implementation Notes for a Rust Port

Source tree: `_references/usearch/` at upstream version `2.25.1` (see
`_references/usearch/VERSION` and `include/usearch/index.hpp:10-12`).

This document is the input to downstream component-design docs. It records
exactly what the C++ implementation does — byte layouts, traversal logic,
locking model, and API surface — without proposing Rust APIs. All file/line
citations are clickable in repo-aware viewers; treat them as load-bearing.

Scope reminder: only the dense, equi-dimensional `index_dense_gt` path is
documented (the only one used by the C99 ABI and the cxx Rust binding). The
joins/clustering paths and the sparse / set-similarity templates are out of
scope here.

## 1. Wire format byte layout — the contract

### 1.1 Two-class layered design

A serialized USearch dense index is **three concatenated regions**, in this
order on disk
(`include/usearch/index_dense.hpp:377-386`):

```
[ vectors blob ]  [ 64-byte dense head ]  [ HNSW graph blob ]
       |                  |                       |
       v                  v                       v
  optional, ".bbin"   index_dense_head_t   index_serialized_header_t
                      (this section)       + per-node bytes
                                           (Section 2)
```

- The **vectors blob** can be omitted when `serialization_config_t::exclude_vectors == true`
  (`include/usearch/index_dense.hpp:170-173`, `1062-1088`). When present it starts
  with a 2-element matrix-shape header (rows, byte-cols) and is followed by a
  flat row-major matrix of raw quantized vector bytes.
- The **64-byte dense head** is the magic-bearing region used to identify the
  file and to recover all type information needed to wire up a metric on load.
- The **HNSW graph blob** is the graph topology, written by the underlying
  type-erased `index_gt::save_to_stream` (`include/usearch/index.hpp:3411-3452`).

The split is deliberate. The dense head carries no graph hyper-parameters
(`connectivity` is in the inner `index_serialized_header_t`); the inner
graph header carries no scalar/metric info. Both are needed to fully reconstruct
the index.

### 1.2 The vectors blob

Generated by `index_dense_gt::save_to_stream` at
`include/usearch/index_dense.hpp:1062-1088`:

| Off (32-bit hdr) | Off (64-bit hdr) | Bytes | Meaning                                                |
|------------------|------------------|-------|--------------------------------------------------------|
| `0x00`           | `0x00`           | 4 / 8 | `matrix_rows` (= number of nodes)                      |
| `0x04`           | `0x08`           | 4 / 8 | `matrix_cols` (= `metric_.bytes_per_vector()`)         |
| `0x08`           | `0x10`           | rows × cols | flat matrix of raw quantized vector bytes        |

The dimension header switches between 32-bit and 64-bit based on
`serialization_config_t::use_64_bit_dimensions` (`index_dense.hpp:170-173`,
`1064-1080`). Default is 32-bit. The matrix is row-major, with each row being
exactly `metric_.bytes_per_vector()` bytes — no padding between rows
(`index_dense.hpp:1083-1087`).

`bytes_per_vector()` is `divide_round_up<8>(dimensions * bits_per_scalar(scalar_kind))`
(`include/usearch/index_plugins.hpp:2978-2980`). For `b1x8_k` this packs 8
dimensions into 1 byte; for `e5m2_k`/`e4m3_k`/`i8_k`/`u8_k` it is `dim` bytes;
for `f16_k`/`bf16_k` it is `2 * dim`; for `f32_k` it is `4 * dim`; for `f64_k`
it is `8 * dim` (`include/usearch/index_plugins.hpp:250-273`).

### 1.3 The 64-byte dense head — `index_dense_head_t`

Structure declared at `include/usearch/index_dense.hpp:42-79`. The buffer is a
fixed `index_dense_head_buffer_t = byte_t[64]` and the static_assert at line 31
locks the size. Field-by-field annotation, with offsets computed from the
constructor at `include/usearch/index_dense.hpp:66-78` (each `exchange(ptr, ptr + sizeof(T))`
advances the cursor):

| Offset | Size | Field                  | Type                       | Notes |
|-------:|-----:|------------------------|----------------------------|-------|
|   0x00 |    7 | `magic`                | `char[7]`                  | ASCII `"usearch"`, no NUL. Set in `save_to_stream` at line 1095 via `std::memcpy(buffer, default_magic(), strlen("usearch"))`. The 7th byte is the literal `'h'`; byte 7 is left at zero by the `memset` at line 1093. |
|   0x07 |    2 | `version_major`        | `uint16_t` (LE)            | Must equal `USEARCH_VERSION_MAJOR` (currently `2`, `index.hpp:10`). Mismatch → "File format may be different, please rebuild" (`index_dense.hpp:1198`, `1314`). |
|   0x09 |    2 | `version_minor`        | `uint16_t` (LE)            | Currently `25` (`index.hpp:11`). Pre-2.10 indexes get re-mapped scalar codes via `fix_pre_2_10_metadata` (`index_dense.hpp:239-247`). |
|   0x0B |    2 | `version_patch`        | `uint16_t` (LE)            | Currently `1` (`index.hpp:12`). |
|   0x0D |    1 | `kind_metric`          | `metric_kind_t` (`uint8_t`)| Char-coded enum: `'i'`=ip, `'c'`=cos, `'e'`=l2sq, `'p'`=pearson, `'h'`=haversine, `'d'`=divergence, `'b'`=hamming, `'t'`=tanimoto, `'s'`=sorensen, `'j'`=jaccard, `0`=unknown (`index_plugins.hpp:114-133`). |
|   0x0E |    1 | `kind_scalar`          | `scalar_kind_t` (`uint8_t`)| Numeric enum, see §1.4. Driver of `metric_punned_t::builtin` lookup on load. |
|   0x0F |    1 | `kind_key`             | `scalar_kind_t` (`uint8_t`)| Type used for `vector_key_t`. For `index_dense_t` (default key `uint64_t`) this is `u64_k=14`. Mismatch → "Key type doesn't match, consider rebuilding" (`index_dense.hpp:1202-1203`, `1318-1319`). |
|   0x10 |    1 | `kind_compressed_slot` | `scalar_kind_t` (`uint8_t`)| Type used for slot IDs. For `index_dense_t` this is `u32_k=15`; `index_dense_big_t` uses `u40_k=2` (`index_dense.hpp:2294-2295`). Mismatch → "Slot type doesn't match, consider rebuilding" (`index_dense.hpp:1204-1205`, `1320-1321`). |
|   0x11 |    8 | `count_present`        | `uint64_t` (LE)            | `size()` excluding tombstones. |
|   0x19 |    8 | `count_deleted`        | `uint64_t` (LE)            | Deleted slots present in the graph but not user-visible. |
|   0x21 |    8 | `dimensions`           | `uint64_t` (LE)            | Logical scalar count, **not bytes**. `bytes_per_vector` is computed on load. |
|   0x29 |    1 | `multi`                | `bool` (1 byte)            | When true, the same key may map to multiple vectors. Determines whether `add_` rejects duplicates (`index_dense.hpp:2065`). |
|   0x2A |    6 | reserved               | zero                       | The header docstring at line 38-40 says "leaving 5 bytes at the end vacant"; in practice the `bool` plus `memset(...,0,64)` (line 1093) gives 6 zero bytes (offsets 0x2A..0x2F). |

**Endianness**: every multi-byte field is read/written via
`misaligned_load`/`misaligned_store` (`include/usearch/index.hpp:201-211`),
which `memcpy` from/to a host-endian scalar. There is no byte-swapping; the
file format is implicitly **little-endian** (USearch is built/tested only on
LE x86-64 / ARM64). A big-endian Rust port must swap on load/save.

**Alignment**: the `index_dense_head_t` buffer is just 64 bytes treated as
unaligned — every field is accessed via `misaligned_ref_gt`
(`index_dense.hpp:50-64` and `index.hpp:248-267`). The actual cursor inside the
buffer is *not* naturally aligned for `uint64_t` (`count_present` starts at
offset 0x11), which is why the misaligned-ref machinery exists.

### 1.5 The HNSW graph blob — `index_serialized_header_t`

Written by `index_gt::save_to_stream` at `include/usearch/index.hpp:3411-3452`.
Begins with a 40-byte fixed header (`include/usearch/index.hpp:1990-1996`):

| Offset | Size | Field | Notes |
|-------:|-----:|-------|-------|
| 0x00 | 8 | `size` | number of node entries (== `matrix_rows` in vectors blob) |
| 0x08 | 8 | `connectivity` | `M` from the HNSW paper; default 16 (`index.hpp:1467`) |
| 0x10 | 8 | `connectivity_base` | `M0`; default `2*M` = 32 (`index.hpp:1495`) |
| 0x18 | 8 | `max_level` | `level_t` of entry node |
| 0x20 | 8 | `entry_slot` | slot ID of the unique top-level node |

After the 40-byte header come **two streamed blocks**
(`index.hpp:3430-3449`):

1. A flat array of `size` levels: `level_t[size]` where `level_t` is
   `std::int16_t` (2 bytes; `index.hpp:2207`). Slot `i` at this level is the
   tallest level in which node `i` participates.
2. Concatenated **node tapes**, one per slot. Layout per node is described in
   §2.

The reason the levels precede the node bodies is that *the body size for slot
`i` is a function of `levels[i]`* — see `node_bytes_(level)` at
`index.hpp:3877-3879`. Holding all levels first lets `view()` precompute every
node's byte offset before reading any body
(`index.hpp:3675-3680`).

### 1.6 Per-node tape layout

A `node_t` is just a pointer to a contiguous tape (`include/usearch/index.hpp:2243-2264`).
The byte layout of one node tape, for a node that lives in levels `0..L` (so
participates in `L+1` levels):

```
+----------------+----------------+--------------------------------+--------------------------------+ ... +--------------------------------+
| key (K bytes)  | level (2 bytes)| base level slab (M0_bytes)     | level-1 slab  (M_bytes)        |     | level-L slab  (M_bytes)        |
+----------------+----------------+--------------------------------+--------------------------------+ ... +--------------------------------+
^                ^                ^                                ^                                      ^
0                K               K+2                              K+2+M0_bytes                            K+2+M0_bytes+(L-1)*M_bytes
```

Key facts:

- **Key width** `K` = `sizeof(vector_key_t)` (8 for the default, 16 for the
  `index_dense_big_t` UUID variant; see `node_head_bytes_` at `index.hpp:2212`).
- **Level** is `int16_t` (`level_t` typedef, `index.hpp:2207`) and is always
  >= 0 in practice. It is stored at offset `K`. Note this differs from the
  in-stream `levels[]` array preceding the node bodies — that array is a
  separate copy used only to size-and-locate the bodies during `view`/`load`.
- Each **per-level slab** starts with a `neighbors_count_t = uint32_t`
  (`index.hpp:2206`, `2275-2306`) holding the current number of neighbors in
  that slab, followed by `cap_at_level` slots of `compressed_slot_t`. The slab
  is **sized to the cap, not the count** — every slab is padded out to the
  worst case so neighbor lists can grow in place during construction without
  reallocation.
- **Slab capacities**:
  - Base layer slab cap = `connectivity_base` (default 32).
  - Higher-layer slab cap = `connectivity` (default 16).
- **Slab byte sizes** are precomputed in `precompute_` at `index.hpp:3866-3872`:
  - `neighbors_base_bytes = connectivity_base * sizeof(slot) + 4`
  - `neighbors_bytes      = connectivity      * sizeof(slot) + 4`
- **Total node body size** for level `L`:
  `node_bytes_(L) = node_head_bytes_() + neighbors_base_bytes + L * neighbors_bytes`
  (`index.hpp:3877-3879`, `3881-3883`).

### 1.7 Where `uint40_t` is used

`uint40_t` is a 5-byte little-endian unsigned integer, `pack`-attributed and
memcpy-based (`index.hpp:1094-1158`). It is *only* used as the
**`compressed_slot_t`** type when callers explicitly want a >4-billion-entry
index. The default `index_dense_t` uses `default_slot_t = uint32_t`
(`index.hpp:1999`); the `index_dense_big_t` typedef switches to
`uuid_t` keys + `uint40_t` slots
(`index_dense.hpp:2294-2295`). The wire format reflects this in `kind_compressed_slot`
(byte at offset 0x10 of the dense head): `u32_k = 15` for the default,
`u40_k = 2` for the big variant.

`uint40_t` round-trips via `memcpy` of 5 octets (`index.hpp:1110, 1132`). On
big-endian hosts this would need adjustment — usearch does not handle BE.

### 1.8 Concrete byte diagram — 4 nodes, max-level 2, 8-byte vectors

A toy index with `connectivity=2` (M), `connectivity_base=4` (M0),
`dimensions=8`, `scalar_kind=u8_k`, `metric_kind=l2sq_k`, default keys & slots.
Suppose node 0 is at level 2 (entry), node 1 at level 1, nodes 2 and 3 at level 0.

Per-level neighbor cap → bytes:
- base slab cap `M0=4` → `4 + 4*4 = 20` bytes
- non-base slab cap `M=2` → `4 + 2*4 = 12` bytes
- node head = `8 + 2 = 10` bytes

Node body sizes:
- `node_bytes_(0) = 10 + 20 = 30`
- `node_bytes_(1) = 10 + 20 + 12 = 42`
- `node_bytes_(2) = 10 + 20 + 24 = 54`

Full file (with vectors, 32-bit dim header):

```
Offset  Region                 Bytes  Contents
0x0000  vectors header         8      [u32 rows=4][u32 cols=8]
0x0008  vector 0               8      raw 8 bytes
0x0010  vector 1               8      raw 8 bytes
0x0018  vector 2               8      raw 8 bytes
0x0020  vector 3               8      raw 8 bytes
0x0028  dense head             64     "usearch" + ver 2.25.1 + metric='e' + scalar=17 + key=14 + slot=15 + counts + dim=8 + multi=0 + 6 zero
0x0068  graph header           40     size=4 connectivity=2 connectivity_base=4 max_level=2 entry_slot=0
0x0090  levels[0..3]           8      [i16 2][i16 1][i16 0][i16 0]
0x0098  node-0 tape (level 2)  54     key(8) | level(2)=2 | M0 slab(20) | level-1 slab(12) | level-2 slab(12)
0x00CE  node-1 tape (level 1)  42     key(8) | level(2)=1 | M0 slab(20) | level-1 slab(12)
0x00F8  node-2 tape (level 0)  30     key(8) | level(2)=0 | M0 slab(20)
0x0116  node-3 tape (level 0)  30     key(8) | level(2)=0 | M0 slab(20)
EOF     0x0134
```

A single per-level slab (e.g. node-0's base-layer slab at offset
`0x0098 + 8 + 2 = 0x00A2`) looks like:

```
0x00A2  uint32 count_neighbors      = 3      (e.g. node-0 connected to nodes 1,2,3)
0x00A6  uint32 slot_id_0            = 1
0x00AA  uint32 slot_id_1            = 2
0x00AE  uint32 slot_id_2            = 3
0x00B2  uint32 slot_id_3 (unused)   = 0      (slab padded to cap=4)
```

`size()` for a slab is read from the count field; the cap-padded tail is
preallocated but not "live" until `push_back` increments the count
(`index.hpp:2294-2306`).

## 2. Graph topology and addressing

### 2.1 Slot → node-pointer mapping

`index_gt::nodes_` is a `buffer_gt<node_t>` indexed by `compressed_slot_t`
(`index.hpp:2407`). A `node_t` is just an opaque pointer-to-tape
(`index.hpp:2243-2264`). Therefore:

- **Slot ID is the array index into `nodes_`.** It is monotonic from 0 up to
  `nodes_count_ - 1`. Slots are reused only after a key has been removed — see
  the "free key" mechanic in §3.5 below.
- **Slot ID is *not* a byte offset.** The byte offset depends on each node's
  level and is computed lazily during traversal via `nodes_[slot].tape()`
  (`index.hpp:2920` and similar), or precomputed for `view()`
  (`index.hpp:3676-3680`).

### 2.2 Per-level neighbor list addressing

Inside a node tape, the neighbor slabs sit immediately after the head:

- `node.neighbors_tape() = tape + node_head_bytes_()` (`index.hpp:2249`).
- Base-layer slab at `neighbors_base_(node) = node.neighbors_tape() + 0`
  (`index.hpp:3921`).
- Level-`L` slab (`L >= 1`) at
  `node.neighbors_tape() + neighbors_base_bytes + (L-1) * neighbors_bytes`
  (`index.hpp:3923-3926`).
- The two are dispatched by the public helper
  `neighbors_(node, level)` at `index.hpp:3928-3930`.

Inside a slab, the `neighbors_count_t` count occupies the first 4 bytes and
slots begin at offset 4 (`neighbors_ref_t::shift` at `index.hpp:2278-2280`).
Slots are stored as raw `compressed_slot_t` values, and reads/writes go through
the misaligned-ptr/ref helpers because a `uint32_t` slot can fall on a non-4
byte boundary when `key_t` is 8 bytes and the level (2 bytes) precedes the
slab.

### 2.3 The entry point

A single `entry_slot_` (`index.hpp:2402`) and `max_level_` (`index.hpp:2399`)
form the entry to the graph. Both are protected by `global_mutex_`
(`index.hpp:2396`). They are written:

- On first insert: at `index.hpp:2966-2969` (when `old_size == 0`).
- On insert of a new tallest node:
  `if (new_target_level > max_level_copy)` → `entry_slot_ = new_slot; max_level_ = new_target_level;`
  (`index.hpp:3005-3009`).

The entry point is **persisted as `index_serialized_header_t::entry_slot` /
`max_level`** (`index.hpp:3421-3422` write, `3502-3503` read, `3696-3697` view).

### 2.4 `view()` reconstructs topology without copying

Walkthrough of `index_gt::view` at `include/usearch/index.hpp:3638-3707`:

1. `reset()` clears prior state (3644).
2. mmap the file (3646; ultimately `mmap()` on Linux/macOS at
   `index.hpp:1948`, `MapViewOfFile` on Windows at `index.hpp:1919`).
3. `memcpy` the 40-byte graph header out of the mapping (3654).
4. Read & validate `connectivity` / `connectivity_base`; recompute slab sizes
   via `precompute_` (3668-3674).
5. **Build offset table**: walk the levels[] array (treated as a misaligned
   pointer into the mmap) and exclusive-scan the per-node body sizes:
   ```cpp
   misaligned_ptr_gt<level_t> levels{file.data() + offset + sizeof(header)};
   offsets[0]  = offset + sizeof(header) + sizeof(level_t) * header.size;
   for i in 1..size: offsets[i] = offsets[i-1] + node_bytes_(levels[i-1]);
   ```
   (`index.hpp:3675-3680`). This is the only O(N) step on `view`.
6. Reserve `nodes_` and the thread contexts (3686-3694).
7. Restore `nodes_count_`, `max_level_`, `entry_slot_` (3695-3697).
8. **Wire pointers**: `nodes_[i] = node_t{file.data() + offsets[i]}`
   (3700-3704). No bytes are copied — every read of a key, level, or neighbor
   list now goes through the mmap.
9. Stash the `memory_mapped_file_t` on `viewed_file_` to keep the mapping
   alive (3705).

`is_immutable()` becomes true (`index.hpp:2424` returns `bool(viewed_file_)`),
and `add()` is rejected (`index.hpp:2918`). `node_free_` is a no-op when viewed
(`index.hpp:3911-3914`), preventing the destructor from `munmap`ping bytes the
allocator never owned.

## 3. Search algorithm

Three traversal kernels work the graph: `search_for_one_` (greedy descent),
`search_to_insert_` (beam search reused during insertion), and
`search_to_find_in_base_` (beam search at level 0 with predicate filtering).
All live in the private section of `index_gt` and are reused across `add`,
`update`, `search`, and `cluster`.

### 3.1 `index_gt::search` entry-point — `index.hpp:3142-3207`

Pseudocode of the core path:

```
context = contexts_[config.thread]
top.clear(); next.clear()
expansion = max(config.expansion, wanted)
top.reserve(expansion); next.reserve(expansion)

closest = search_for_one_(query, metric, prefetch, entry_slot_, max_level_,
                          /*end_level=*/0, context)

search_to_find_in_base_(query, metric, predicate, prefetch, closest,
                        expansion, context)
top.sort_ascending()
top.shrink(wanted)
return search_result_t{count = top.size(), ...}
```

**Default values** if the caller passes zero: `wanted` short-circuits to
empty (`index.hpp:3157`); `expansion` falls back to
`default_expansion_search() = 64` (`index.hpp:1477`, `3161-3162`). The actual
beam width used is `max(expansion, wanted)` (`index.hpp:3184`).

If `config.exact == true`, `search_exact_` is used instead — a flat O(N) scan
that respects the predicate and dumps into `top` (`index.hpp:3178-3181`,
`4423-4440`).

### 3.2 `search_for_one_` — greedy zoom-in — `index.hpp:4125-4166`

Used to descend from level `begin_level` down to `end_level`, tracking only the
single closest candidate. Algorithm:

```
visits.clear()
closest_dist = metric(query, closest_slot)
for level in (begin_level .. end_level], descending:
  loop:
    changed = false
    for each candidate_slot in neighbors(closest_slot, level):  // M-fan-out
      d = metric(query, candidate_slot)
      if d < closest_dist:
        closest_dist = d
        closest_slot = candidate_slot
        changed = true
    if !changed: break
```

This deliberately **does not** maintain a `next`/`top` queue — at upper levels
M is small (default 16) and a single greedy walk is cheaper than priority-queue
maintenance. The visited-set (`visits_hash_set_t`) is reused across iterations
inside this routine but is not propagated into the base-layer search.

Locking: it acquires `optional_node_lock_` per visited node when the index is
mutable (`index.hpp:4137`, `4143`); on a viewed/immutable index locking is
skipped entirely (the `need_lock` test).

### 3.3 `search_to_find_in_base_` — beam search at level 0 — `index.hpp:4347-4418`

The base-layer beam search, used by both query and (with a lock variant) by
insertion. Shape:

```
visits.clear()
next.clear() (max-heap on -distance; pop-min in disguise)
top.clear()  (sorted-buffer, .top() returns the worst entry)

next.insert({-radius, start_slot})
visits.set(start_slot)
if predicate(start): top.insert({radius, start_slot})

while !next.empty():
    cand = next.top()       // smallest distance still on the frontier
    if -cand.distance > radius and top.size() == top_limit: break
    next.pop()
    for each successor_slot in neighbors(cand.slot, 0):
        if visits.set(successor_slot): continue        // already seen
        d = metric(query, successor_slot)
        if top.size() < top_limit or d < radius:
            next.insert({-d, successor_slot})
            if predicate_ok(successor):
                top.insert({d, successor_slot}, top_limit)
                radius = top.top().distance
```

`top_limit` is set to the caller's `expansion` (`index.hpp:4355`). The base
layer search **does not lock nodes** — it relies on the `need_lock`-controlled
insert path to either be running serially with respect to readers or to
tolerate the eventual-consistency of in-flight neighbor lists (the size header
is updated via misaligned scalar stores, so a torn read is acceptable).

The **min-heap** for the frontier is a `max_heap_gt<candidate_t>` with negated
distances stored (`index.hpp:4198-4199`, `4369`); `next.top()` returns the
largest stored value, i.e. the smallest *real* distance. This keeps the
priority-queue contract single-purpose (max-heap) while reusing the same
template for "best-of" sets stored in `top`.

The **`top` collection** is a `sorted_buffer_gt<candidate_t>` (a sorted-array
data structure, `index.hpp:973-1080`). Its `.top()` returns `elements_[size-1]`,
the *worst* current entry. `insert(elem, limit)` does a `lower_bound` search,
shifts elements to make room, and grows up to `limit` (`index.hpp:1053-1064`).
When the buffer is full, the new element either replaces the worst-and-shifts
or is dropped if it is itself worse than the worst.

Predicate integration: the predicate is evaluated only when deciding whether to
insert into `top` (`index.hpp:4408-4412`). Crucially, **filtered candidates are
still pushed to `next`** — the graph is traversed even through nodes the
predicate rejects, just without recording them. This preserves recall on
selective predicates at the cost of more distance computations.

The dense layer wraps the user predicate to additionally exclude tombstones —
nodes whose key equals `free_key_` (`index_dense.hpp:2138-2148`).

### 3.4 Result extraction

After `search_to_find_in_base_`, `top.sort_ascending()` orders best-first and
`top.shrink(wanted)` truncates (`index.hpp:3199-3200`). The caller pulls
results via:
- `search_result_t::operator[]` (`index.hpp:2761-2766`) — returns `match_t{member_cref_t, distance}`.
- `dump_to(keys, distances)` and overloads (`index.hpp:2805-2868`) — copy out.
- `merge_into` for incremental top-K aggregation (`index.hpp:2777-2797`).

The `top_candidates_t` lives in the per-thread `context_t`; `search_result_t`
holds a pointer back into it (`index.hpp:2719-2725`). On the dense layer the
result also drags a `thread_lock_t` to keep the slot reserved until the result
goes out of scope (`index_dense.hpp:550-564`).

### 3.5 Tombstones

`index_dense_gt::remove` does not actually delete a node — it rewrites the
node's key to `free_key_` (a sentinel; default is `default_free_value<key>()`
i.e. all-1s) and pushes the slot onto `free_keys_` (`index_dense.hpp:1545-1567`,
`507-510`). On the next `add`, `free_keys_.try_pop` reuses that slot via
`index_t::update` instead of `add` (`index_dense.hpp:2078-2114`).

Searches transparently skip tombstones because the dense-layer predicate
wrapper rejects any member whose key matches `free_key_`
(`index_dense.hpp:2138-2148`).

## 4. Insert algorithm

The insert path crosses both layers: `index_dense_gt::add_` handles type
casting + free-slot reuse + key bookkeeping; `index_gt::add` handles the actual
graph mutation.

### 4.1 `index_dense_gt::add_` — `include/usearch/index_dense.hpp:2060-2115`

1. **Duplicate check** (line 2065): if `!multi && enable_key_lookups`, reject
   if the key already exists.
2. **Reserve a thread context** (line 2069) → `thread_lock_t` (line 516-535).
3. **Cast vector to `scalar_kind_`** (lines 2071-2076) using `cast_punned_t`.
   If a cast happens the casted bytes are written into a per-thread slab in
   `cast_buffer_` (line 2072), and `copy_vector` is forced true.
4. **Try to recycle a tombstone slot** (lines 2078-2086): pop one from
   `free_keys_` if available.
5. **Allocate vector storage** if needed (lines 2088-2093) via
   `vectors_tape_allocator_` (an arena-style allocator,
   `index_plugins.hpp:1644-1770`).
6. **Define `on_success` callback** (lines 2094-2105) that:
   - registers the key→slot mapping in `slot_lookup_` under
     `slot_lookup_mutex_` (write lock);
   - copies the casted vector bytes into `vectors_lookup_[slot]` (the
     per-slot raw vector pointer table; line 460).
7. **Call `index_gt::update` if reusing a slot, else `add`** (lines 2112-2114),
   passing `metric_proxy_t` which routes to the punned metric and resolves
   peer vectors via `vectors_lookup_[slot]` (`index_dense.hpp:419-438`).

### 4.2 `index_gt::add` — graph mutation — `include/usearch/index.hpp:2900-3011`

1. **Coerce `expansion`** to `default_expansion_add() = 128` if zero (line
   2913-2915).
2. **Reserve thread-local heaps** (`top` and `next` in the `context_t`):
   `top_limit = max(connectivity_max + 1, expansion)`, `next.reserve(expansion)`.
   The `+1` is a heuristic-overflow slot used by `select_neighbors_heuristic`
   (line 2929-2935).
3. **Choose target level** under `global_mutex_` (line 2938-2941):
   ```cpp
   level_t new_target_level = choose_random_level_(context.level_generator);
   ```
   `choose_random_level_` (line 4055-4059):
   ```cpp
   double r = -log(uniform(0,1)) * inverse_log_connectivity;
   return level_t(r);    // truncates toward zero
   ```
   `inverse_log_connectivity = 1.0 / log(connectivity)` (precomputed at
   `index.hpp:3866-3872`). The RNG is `std::default_random_engine` per-thread
   (`index.hpp:2334`) — implementation-defined PRNG, **not byte-stable across
   stdlibs**.
4. **Atomically reserve a slot** by `nodes_count_.fetch_add(1)` (line 2945).
5. **Allocate node tape** sized to `new_target_level` via
   `node_make_(key, new_target_level)` (line 2952; allocator at line 3885-3901).
   The tape is `memset` to zero so all neighbor counts/lists start empty.
6. If `new_target_level <= max_level_copy`, drop the global lock early
   (line 2957-2958) so other threads can also insert.
7. **Special-case first node** (line 2966-2970): set `entry_slot_` /
   `max_level_` and return — no neighbors to wire.
8. **Greedy descent** through levels above `new_target_level` to find a good
   start node:
   ```cpp
   closest_slot = search_for_one_(value, metric, prefetch,
                                  entry_slot_copy, max_level_copy,
                                  new_target_level, context);
   ```
   (line 2978-2981).
9. **For each level from `min(new_target_level, max_level_copy)` down to 0**
   (line 2984-2995):
   a. `search_to_insert_` runs a beam search of width `expansion`
      starting at `closest_slot`. Output is `top` (sorted-best-first) +
      `next` (frontier). This routine *does* take per-node spinlocks on
      neighbors, since other threads may be linking concurrently.
   b. Under the new node's lock (`node_lock_(new_slot)`),
      `form_links_to_closest_` (line 3980-4001) is called.
   c. `form_reverse_links_` (line 4003-4053) wires the back-edges.
10. **Update entry point** if the new node went above `max_level_copy`
    (line 3005-3009).

### 4.3 The neighbor-selection heuristic — `refine_` — `index.hpp:4467-4522`

USearch implements **only the heuristic variant** from the original HNSW paper
("Algorithm 4: SELECT-NEIGHBORS-HEURISTIC"). There is no
`select_neighbors_simple` flavor — `refine_` is always called when forming
links. The simple "just take the top-M closest" behavior is exposed only as a
trivial early-out: if `top.size() < needed`, the routine returns the existing
top without filtering (line 4488-4491).

The selection logic, in plain English:

```
sort top by ascending distance to query
keep top[0]
for each subsequent candidate c in top:
    accept = true
    for each already-kept submitted s:
        if distance(c, s) < distance(c, query):
            accept = false; break
    if accept: append c to the kept prefix
shrink top to the kept prefix
```

The "diversification" rule is: a candidate is rejected if it's closer to some
already-selected neighbor than it is to the query. This produces an isotropic
neighborhood — the kept neighbors point in different "directions" from the new
node, improving graph connectivity.

`refine_` accepts an optional `override_slot` / `override_value` pair (line
4479-4485) used by `update`: when computing `inter_neighbor_distance_`, a
specific slot's stored vector is replaced with the supplied value. This lets
`update` use the *new* vector for inter-neighbor diversification while the old
vector is still in `vectors_lookup_`.

The submitted-counter of distance evaluations is added to
`refines_counter` per call (lines 4501-4511) and propagated into
`add_result_t::computed_distances_in_refines` (line 2999-3000) for diagnostics.

### 4.4 Forming forward links — `form_links_to_closest_` — `index.hpp:3980-4001`

```
top_view = refine_(metric, /*needed=*/connectivity, top, ...)   // shrinks `top`
new_neighbors = neighbors(new_node, level)
for c in top_view:
    new_neighbors.push_back(c.slot)
return top_view
```

`needed = connectivity` regardless of level — i.e. the per-level cap is the
*non-base* connectivity even for the base layer. This may surprise readers:
the *cap* of the base slab is `connectivity_base` (M0), so the slab is sized
to fit M0 neighbors, but `form_links_to_closest_` only ever inserts up to
`connectivity` (M) forward links. The extra capacity in the base slab is
filled in via reverse-link refinement from later inserters — see §4.5.

### 4.5 Forming reverse links — `form_reverse_links_` — `index.hpp:4003-4053`

For each forward neighbor `close_slot` of the new node:

1. Acquire that node's spinlock.
2. **Cheap path**: if `close_header.size() < connectivity_max`, just
   `push_back(new_slot)` on its slab and continue. `connectivity_max` is
   `connectivity` for non-base levels and `connectivity_base` for level 0
   (line 4009).
3. **Refinement path** (slab is full):
   a. Build `top_for_refine` containing `(distance(value, close_slot), new_slot)`
      plus all existing successors with distances.
   b. `close_header.clear()`.
   c. Call `refine_(metric, connectivity_max, top_for_refine, ...)`.
   d. Push the refined neighbors back into the slab.

This is what fills base slabs out to the M0 cap over time: as new nodes wire
themselves into existing nodes' neighborhoods, the heuristic reshuffles the
existing slab to keep the best M0 neighbors.

### 4.6 The striped-lock concurrency story

There is **no per-node mutex**. Instead `striped_locks_gt` at
`include/usearch/index.hpp:600-717` provides a fixed-size array of cache-line-padded
spin-flags (`atomic<uint8_t>` with 128-byte padding, `index.hpp:616-620`) and
maps every slot to one stripe via Fibonacci hashing:

```cpp
stripe_for(slot) = (slot * 0x9E3779B97F4A7C15ull) >> shift
                                                       ^
                                                       chosen so 1 << (64-shift) == count
```

(`index.hpp:633-635`). The number of stripes is `ceil2(threads * connectivity * 4)`
clamped to >= 256 (`index.hpp:656-660`); shift is set so stripe count is a
power of two and the shift selects the high bits of the Fibonacci product.

Operations:
- `lock(slot)`: spin on `atomic_set` (xchg-then-test) until success
  (`index.hpp:711-714`); yields between attempts.
- `node_lock_t` RAII wrapper auto-unlocks (`index.hpp:3932-3941`).

**What this actually protects**: the *neighbor slab* of a node. When two
threads are inserting and each wants to mutate node X's slab, they collide
on the stripe for X. They **don't collide** on per-node fields like `key` or
`level`, which are only ever set at allocation time and never mutated post-insert
(barring `update`'s key rewrite, which is unprotected and explicitly noted as
non-transactional, `index.hpp:3017-3018`).

**What this does *not* protect**: read-only traversal at search time. The base
layer search (`search_to_find_in_base_`) does not take stripe locks at all
(`index.hpp:4347-4418`); it just races against in-flight inserters and tolerates
torn slab reads — neighbor counts can be observed mid-update, but the slab is
zero-padded so an over-read returns slot 0 (which is always a valid live node)
and a torn slot read returns either old or new bytes (both valid). Inserters
inside `search_to_insert_` *do* lock candidates (`index.hpp:4215`) because they
also dereference into peer slabs.

Two more locks of note:
- `global_mutex_` (`index.hpp:2396`) protects `max_level_` / `entry_slot_`
  during entry-point upgrades. Held for the level-pick + slot-allocation
  prelude of every `add`.
- `nodes_mutexes_` (the striped pool) is sized at `reserve()` time
  (`index.hpp:2638`). Stripes do **not** scale with the index — only with
  `threads * connectivity_max`, so memory overhead is bounded.

In the dense wrapper, the per-thread context pool is itself rate-limited by
`available_threads_` (`index_dense.hpp:466-469`): `thread_lock_` pops a
thread ID off this ring under `available_threads_mutex_` (`2050`), and the
RAII wrapper pushes it back on destruction (`2053-2058`). This is what
enforces the `threads_add` / `threads_search` limits at the wrapper level.

## 5. SIMD distance kernel organization

### 5.1 The dispatcher — `metric_punned_t`

Defined at `include/usearch/index_plugins.hpp:2823-3300+`. The class holds a
member-function pointer `metric_routed_t` (line 2840) plus an opaque
function pointer (`metric_ptr_`) and an integer side-state
(`metric_third_arg_`, line 2842). `operator()` invokes the routed pointer
(line 2858-2860), which in turn calls one of:

- `invoke_array_array` (line 3070-3074): two-argument metric callback;
- `invoke_array_array_third` (line 3065-3069): three-argument with state;
- `invoke_numkong<accumulator, reverse_ip>` (line 2998-3004): typed wrapper
  around the NumKong dense kernel signature.

### 5.2 Dispatch is **runtime**

Construction goes through `metric_punned_t::builtin(dimensions, metric_kind, scalar_kind)`
(`index_plugins.hpp:2883-2898`). The builder:

1. Defaults `metric_routed_ = invoke_array_array_third`.
2. Calls `configure_with_numkong()` first; if that returns false, falls
   through to `configure_with_autovec()` (line 2894-2895).

`configure_with_numkong` at `index_plugins.hpp:3012-3061` looks up an actual
SIMD kernel from the NumKong runtime, picking the best ISA available
(`nk_find_kernel_punned`). It maps:

- `metric_kind_t::ip_k` → `nk_kernel_dot_k`
- `metric_kind_t::cos_k` → `nk_kernel_angular_k`
- `metric_kind_t::l2sq_k` → `nk_kernel_sqeuclidean_k`
- `metric_kind_t::hamming_k` → `nk_kernel_hamming_k`
- `metric_kind_t::tanimoto_k` / `jaccard_k` → `nk_kernel_jaccard_k`

Other metric kinds (`pearson`, `haversine`, `divergence`, `sorensen`) fall
through and use the autovec backend.

`configure_with_autovec` (line 3075+) picks a function pointer from a giant
switch on `(metric_kind, scalar_kind)`, instantiating templates such as
`metric_l2sq_gt<f32_t>` or `metric_cos_gt<f16_t, f32_t>`. The two-template-
parameter forms specify the input scalar and the accumulator type — so for
low-precision inputs the math is done in `f32_t`.

The autovec kernels live in `include/usearch/index_plugins.hpp` around
lines 2219-2780 and are written as ordinary loops with `#pragma omp simd` /
`#pragma clang loop vectorize(enable)` / `#pragma GCC ivdep` hints
(e.g. `metric_ip_gt::operator()` at lines 2219-2236). They are explicitly
**not hand-written intrinsics** — usearch relies on autovectorization plus
NumKong for ISA-specific code.

### 5.3 Build-time NumKong is **off by default**

`USEARCH_USE_NUMKONG` defaults to 0 (`index_plugins.hpp:25-27`). When the
macro is 0, `configure_with_numkong()` is a stub that returns false
(`index_plugins.hpp:3062-3064`), so the autovec path is the only one available.
Production builds set `-DUSEARCH_USE_NUMKONG=1` (see the bench docs at
`BENCHMARKS.md:65-67`).

## 6. Determinism caveats

### 6.1 Sources of non-determinism

The build is **not deterministic** by default. The major sources:

1. **Per-thread RNG state.** Level assignment uses
   `std::default_random_engine` stored on the per-thread `context_t`
   (`index.hpp:2334`). Each context's generator is default-constructed (same
   seed across threads), but **the order in which contexts produce numbers
   depends on insertion order × thread scheduling**.

2. **Insert-order sensitivity.** Even single-threaded, `add` reads the
   *current* `entry_slot_` and `max_level_` (line 2939-2940) and uses them as
   the descent root. Two indexes built from the same vectors in different
   order will have different entry points and different greedy paths, hence
   different graphs.

3. **Multi-threaded reverse-link refinement.** `form_reverse_links_` runs
   `refine_` on full slabs to choose which neighbors to keep. Other threads
   can be concurrently mutating those slabs (under stripe locks). The order
   of arrivals at a saturated slab determines which `refine_` invocation
   "wins" the final composition.

4. **Free-slot reuse ordering.** When tombstones exist, the next `add` reuses
   whichever slot pops first off `free_keys_` (a ring queue,
   `index_dense.hpp:507`, `2079-2086`). Order depends on prior `remove` order.

5. **PRNG implementation-defined.** `std::default_random_engine` is
   implementation-defined; the same seed produces different streams under
   libstdc++ vs libc++ vs MSVC's STL.

### 6.2 `expansion_add` mechanics

`expansion_add` is **not** the queue size at the upper levels — those use the
fixed greedy descent (`search_for_one_`) with no queue. It is the `top_limit`
passed to `search_to_insert_` at every level the new node participates in
(`index.hpp:2986`). Concretely:

- `search_to_insert_` reserves `top` and `next` to size `expansion`
  (`index.hpp:4189-4192`).
- The beam keeps expanding until either `next` is empty, or the smallest
  un-expanded distance exceeds `radius` (the worst-of-top) **and** `top` is
  full (line 4207-4208).
- Larger `expansion` means more candidates evaluated → better graph quality →
  slower insert.

The default is 128 (`index.hpp:1472`); FAISS uses 40, hnswlib uses 200
(`BENCHMARKS.md:13-17`).

### 6.3 Forcing reproducible builds

To get bit-stable graphs across runs, a Rust port must:

1. **Single-threaded insert.** Pin `add` to one logical thread; do not call
   it concurrently. The dense wrapper enforces this only as a thread-pool
   limit, not a sequencing guarantee.
2. **Sorted/canonical input order.** The order vectors are added determines
   the structure. Sort by key or by some content hash before insertion.
3. **Deterministic RNG.** Replace the `std::default_random_engine` with a
   fixed-seed PRNG of known specification (e.g. PCG, ChaCha) and seed it
   identically per build.
4. **Disable tombstone reuse.** Either never call `remove`, or rebuild from
   scratch when reproducing a snapshot.
5. **Same `connectivity` / `connectivity_base` / `expansion_add`** across
   producer and consumer. These are persisted in the file
   (§1.5), so loaders will read them back; reproducing a build requires
   matching them on the producer side.

Even with all five, the **distance kernel** must produce identical output
bit-for-bit. Floating-point reductions inside `metric_l2sq_gt` (etc.) sum
in source-order with `#pragma omp simd`; the SIMD-collapsed reduction tree
depends on compiler/ISA. For bit-stable distances, disable autovectorization
on the metric or use a fixed-shape integer kernel.

## 7. Save / load / view code paths

### 7.1 Save

`index_dense_gt::save_to_stream` at `include/usearch/index_dense.hpp:1052-1120`:

1. (Optional) Write the dimensions header (`uint32_t[2]` or `uint64_t[2]`).
2. (Optional) Write `matrix_rows` vectors back-to-back, each
   `metric_.bytes_per_vector()` bytes, pulled from `vectors_lookup_[i]`.
3. Build the 64-byte dense head:
   - `memset(buffer, 0, 64)` (line 1093)
   - `memcpy(buffer, "usearch", 7)` (line 1095)
   - Set version, metric/scalar/key/slot kinds, counts, dimensions, multi.
4. Write the 64-byte head.
5. Delegate to `index_gt::save_to_stream` for the graph header + per-node
   levels[] + per-node bodies.

The streaming output callback is application-supplied. The two file-based
overloads (`save(output_file_t)`, `save(memory_mapped_file_t)`) just wrap a
filesystem write or a `memcpy` into the mmap respectively
(`index_dense.hpp:1381-1427`).

`serialized_length()` (line 1125-1133) computes the exact byte size: header(s)
+ matrix + dense head + graph header + sum of per-node body sizes
(`index.hpp:3401-3406`).

### 7.2 Load (full materialization)

`index_dense_gt::load_from_stream` at `include/usearch/index_dense.hpp:1142-1246`:

1. Save current `limits_` (so threads_add / threads_search are preserved).
2. `reset()` discards all old allocations.
3. Read dimensions header.
4. Allocate `vectors_lookup_` and load each vector via
   `vectors_tape_allocator_.allocate(matrix_cols)` followed by `input(...)`.
5. Read the 64-byte dense head; verify magic, version_major, key kind, slot
   kind. Apply `fix_pre_2_10_metadata`.
6. `metric_ = metric_t::builtin(head.dimensions, head.kind_metric, head.kind_scalar)`
   reconstructs the SIMD metric.
7. Allocate the inner `index_t` and call `typed_->load_from_stream`.
8. Validate that `typed_->size() == matrix_rows`.
9. Re-reserve thread contexts and rebuild `slot_lookup_` / `free_keys_` via
   `reindex_keys_()` (line 1244, body at 2223-2253).

`index_gt::load_from_stream` at `include/usearch/index.hpp:3457-3517`:

1. Read the 40-byte header.
2. Read the `levels[]` array.
3. For each node, allocate a tape sized to `levels[i]` and read the body
   bytes. Wrap as a `node_t`.

### 7.3 View (mmap, no copy)

Already described in §2.4 above. Key differences from `load`:

- The dense layer's `view` (`index_dense.hpp:1257-1371`) memcpys the dense
  header and matrix-shape but **does not copy vectors** — `vectors_lookup_[i]`
  is set to `vectors_buffer.data() + matrix_cols * i` (line 1356-1357).
- `index_gt::view` (already covered) builds the offset table and points
  `nodes_[i]` directly into the mmap (`index.hpp:3700-3704`).
- The mmap handle is moved into `viewed_file_` so it stays alive
  (`index_dense.hpp:1342` calls `typed_->view(std::move(file), ...)` which
  ultimately stores it on the inner `viewed_file_` at `index.hpp:3705`).
- `is_immutable()` returns true thereafter; `add`/`update` are rejected.

### 7.4 Forward compatibility

The version check is **strict on major**: `head.version_major !=
USEARCH_VERSION_MAJOR` is a hard error (`index_dense.hpp:1198-1199`,
`1314-1315`). A v3 file cannot be opened by a v2 binary. Within a major:

- v2.x with minor < 10 gets metadata-fixed via `fix_pre_2_10_metadata`
  (`index_dense.hpp:239-247`).
- Other minor/patch differences are accepted silently.

There are no length/version fields inside the per-node bodies, so any future
change to the body layout (e.g. inline timestamps) must bump the major.

### 7.5 Metadata-only reads

Two helpers extract just the dense head without committing to a full load:

- `index_dense_metadata_from_path(path)` at `include/usearch/index_dense.hpp:253-319`.
- `index_dense_metadata_from_buffer(file, offset)` at lines 324-369.

Both probe the file: they first try interpreting bytes 0..63 as the dense head
(if the magic is there, the file was saved with `exclude_vectors=true`);
otherwise they try interpreting the first 8 bytes as a 32-bit dimensions
header, computing where the head should be, and re-checking the magic;
otherwise the same with 64-bit dimensions. This is the implementation of the
"can store vectors externally" feature.

### 8.2 What a wire-compatible Rust port must expose

The task spec specifies wire-compatibility, *not* C99 ABI compatibility. So the
Rust port doesn't need to produce a `.so` with the `usearch_*` symbols. It
needs:

1. **Round-trip for the byte layout in §1**: dense head + graph header +
   levels[] + node bodies + (optional) vectors blob, with bit-identical
   semantics for every `index_dense_head_t` field.
2. **Read/write of every `(metric_kind, scalar_kind, key_kind, slot_kind)`
   tuple it claims to support**. The minimum the upstream tests cover is
   `(l2sq | cos | ip | hamming, f32 | f16 | bf16 | i8 | u8 | b1x8, u64, u32)`.
3. **Honest version reporting**. Major must match the upstream's at the time
   the file was written (currently `2`). Minor/patch can diverge.
4. **Honest pre-2.10 fix-up**. If the port reads files produced before v2.10,
   it must apply `convert_pre_2_10_scalar_kind` to the three kind bytes
   (`index_dense.hpp:214-247`).
5. **Same default values** for `connectivity` (16), `connectivity_base` (32),
   `expansion_add` (128), `expansion_search` (64). These are persisted in
   `index_serialized_header_t`; mismatch on the producer side gives
   non-interoperable indexes (you'll get a different graph) but the file
   itself remains parseable.
6. **Proper handling of the misaligned scalar reads** — `uint64_t` and
   `uint32_t` fields inside the dense head and node tapes are not naturally
   aligned; a Rust port must use `read_unaligned` / `from_le_bytes` and
   never type-pun via `&u64`.

### 9.3 Caveats not yet covered

- **Joins** (`include/usearch/index.hpp:4549+`) and clustering
  (`index_plugins.hpp:3442+`) use the same wire format but have independent
  call paths. Out of scope for this doc.
- **The Python/Java/Swift/Go bindings** add their own metadata layers
  (e.g. Python's keys-as-numpy translation). They all sit on top of the same
  C99 ABI and the same wire format documented here.
- **`enable_key_lookups = false`** runtime mode is *not* persisted in the file
  (noted at `index_dense.hpp:127-129`). A Rust port can ignore it for
  wire-format work; it only affects in-memory lookup overhead.
